# Agent Handoff Template

이 문서는 `지옥` 프로젝트에서 여러 에이전트가 같은 작업을 이어서 처리할 때 사용하는 공통 인계 템플릿입니다.

목적은 다음과 같습니다.

- 다음 담당자가 원문 대화를 다시 찾지 않아도 현재 상태를 이해할 수 있게 합니다.
- 확정사항, 가정사항, 미해결 항목을 분리해 역할 충돌을 줄입니다.
- 어떤 담당자가 왜 이어받아야 하는지 명확히 남깁니다.
- 역할별 판단 범위를 넘지 않도록 작업 상태를 문서형으로 고정합니다.

---

## 1. 현재 작업 범위

- 이번 작업이 무엇을 대상으로 했는지 적습니다.
- 단순 요약이 아니라, 다음 담당자가 바로 이어서 작업할 수 있을 정도로 현재 범위와 맥락을 남깁니다.
- 화면 단위인지, 기능 단위인지, 정책 단위인지, 구조 단위인지, 시각 단위인지 분리해서 적습니다.

예시 형식:

- 이번 작업은 `어느 화면 / 어느 기능 / 어느 정책 / 어느 시각 영역`을 기준으로 진행했다.
- 이번 결과는 `기획 정리 / 구조 설계 / 구현 반영 / 문구 조정 / 데이터 재구성 / 시각 디자인 / 보안 정책 정리 / 백엔드 구조 설계 / 검수` 중 어디까지 반영했는지를 기준으로 한다.
- 다음 담당자가 원문을 다시 찾지 않아도 이해할 수 있도록, 현재 상태와 남은 작업을 항목별로 남긴다.

---

## 2. 이번 작업 담당자

- 이번 작업을 실제로 수행한 담당자를 적습니다.
- 필요한 경우, 이전 담당자와 현재 담당자를 분리해서 적습니다.

예시 형식:

- 이전 담당자: `Hell-Planner`
- 현재 담당자: `Hell-Architect`

또는

- 현재 담당자: `Hell-Visual Designer`

---

## 3. 작업 대상

- 어떤 화면, 기능, 구조, 정책, 데이터, 시각 자산, API를 대상으로 했는지 적습니다.
- 파일 단위와 서비스 단위를 함께 적어도 됩니다.

예시 형식:

- 메인 화면
- 프로필 탐색 목록
- 구원 후보함
- 제안 발송 흐름
- 수락 전/후 정보 공개 정책
- 인증 기업 열람 제한 구조
- 반응형 카드 위계
- 시각 자산 (로고/아이콘/배지)
- API 응답 구조

---

## 4. 변경한 파일 목록

- 실제 수정한 파일만 적습니다.
- 수정하지 않은 파일은 적지 않습니다.
- 코드 파일, 문서 파일, 로그 파일을 구분해 적어도 됩니다.

예시 형식:

### 코드 / 구조 파일
- `src/pages/...`
- `src/components/...`

### 문서 파일
- `docs/planning/...`
- `agents/...`

### 로그 / 보드
- `logs/issue-board.md`
- `logs/error-log.md`
- `logs/unknowns-log.md`

---

## 5. 현재 이해한 조건

- 작업할 때 전제로 삼은 조건을 적습니다.
- 사용자가 직접 말한 요구사항, 기존 결정사항, 현재 프로젝트 핵심 방향을 섞지 말고 정리합니다.
- 이 항목은 다음 담당자가 왜 이런 방향으로 이어받아야 하는지 이해하는 기준이 됩니다.

예시 형식:

- 서비스는 채용공고 중심이 아니라 구직자 프로필 중심의 역방향 구인 플랫폼이다.
- 메인 화면은 소개용 랜딩처럼 보이면 안 되고 운영형 허브처럼 보여야 한다.
- 인증 기업만 관련 직무 범위 안에서 프로필 열람이 가능해야 한다.
- 수락 전과 수락 후의 공개 범위는 다를 수 있다.

---

## 6. 유지해야 할 결정사항

- 이미 확정된 내용만 적습니다.
- 다음 담당자가 임의로 뒤집으면 안 되는 결정사항을 남깁니다.
- 표현/기획/구조/시각/정책/보안/데이터/API 중 무엇인지 구분되게 적습니다.

예시 형식:

- 서비스는 `직장을 올리는 사이트`가 아니라 `구직자가 자기 프로필을 올리는 사이트`다.
- 기업은 사람을 먼저 탐색하고 적합한 사람에게 `구원 제안`을 보낸다.
- 구직자는 제안을 수락한 뒤 직장 정보를 확인하는 흐름을 유지한다.
- `구원 후보함` 명칭은 현재 유지한다.
- 아무 사용자나 자기소개서를 열람할 수 없도록 기업 인증 구조를 둔다.
- 구직자 프로필 노출 범위는 희망 직무 관련 인증 기업만을 기본 방향으로 유지한다.
- 메인 화면은 소개형 콘텐츠보다 실제 상태, 최근 활동, 객체, 액션이 먼저 보여야 한다.

---

## 7. 이번 작업에서 실제로 한 일

- 현재 담당자가 실제로 수행한 작업을 적습니다.
- "무엇을 바꿨는지"와 "무엇은 바꾸지 않았는지"를 구분해서 적으면 좋습니다.
- 추상 표현 대신 실제 작업 내용을 적습니다.

예시 형식:

- 목록 화면의 카드 구조를 운영형 허브 기준으로 재배치했다.
- 카드 제목과 상태 라벨을 한 줄 우선 기준에 맞게 축약했다.
- 샘플처럼 보이는 더미 데이터를 실제 운영 객체처럼 보이도록 재구성했다.
- 섹션별 시각 리듬을 시각 자산과 레이아웃 강약으로 구분해 설계했다.
- 접근 제어 정책이 확정되지 않아 상세 공개 범위는 가정으로 남겼다.
- 구조 변경이 필요한 문제는 Builder가 임의 수정하지 않고 Architect 재판단 대상으로 남겼다.
- 시각 자산 문제가 의심되어 Visual Designer 판단이 필요할 수 있음.

---

## 8. 가정으로 남겨둔 부분

- 확정되지 않았지만 작업을 위해 임시 전제로 둔 내용을 적습니다.
- 사실처럼 단정하지 않습니다.
- 반드시 `가정`이라는 성격이 드러나게 씁니다.

예시 형식:

- 제안 알림 전달 채널은 아직 미확정이며, 내부 알림 또는 외부 알림 채널 연동 가능성을 모두 열어둔다.
- 팀장급 또는 실무 책임자까지 열람 권한을 확대할지 미확정이다.
- 외부 채용대행사 허용 여부는 추후 정책 판단이 필요하다.
- 수락 전 기업 정보 공개 범위는 비공개/일부 공개/전체 공개 중 미확정이다.
- 시각 자산 (로고/배지 스타일)은 초기 방향만 잡았고 확정 전이다.

---

## 9. 보안 정책 확인 필요 항목

- 보안/권한/마스킹/열람 제한/로그 관련 미확정 항목을 적습니다.
- Security 담당자가 이어받아야 할 항목을 분리해서 남깁니다.
- 일반 기획 메모처럼 섞지 말고 정책 단위로 적습니다.

예시 형식:

- 열람 가능 대상 범위
- 희망 직무 기반 제한 방식
- 목록/상세/제안 전/수락 후 공개 범위 차이
- 자기소개서 마스킹 수준
- 회사명 / 연락처 / 경력 상세 노출 기준
- 열람 이력 저장 여부와 범위
- 차단 기능 포함 범위
- 비정상 접근 탐지 후보

---

## 10. 백엔드 설계 확인 필요 항목

- API, 엔티티, 상태값, 세션, 알림, 로그 등 백엔드 기준으로 미확정인 항목을 적습니다.
- Backend 담당자가 이어받아야 할 항목을 분리해서 남깁니다.

예시 형식:

- 프로필 목록 응답 필드와 상세 응답 필드 차이
- 수락 전/후 응답 필드 차이
- 인증 상태 enum 정의
- 제안 발송 / 수락 / 거절 상태 전이
- 열람 이력 저장 테이블 또는 로그 구조
- 차단 처리 방식
- 알림 전달 채널 구조
- 프론트가 사용할 상태값 네이밍

---

## 11. 구조적으로 추가 확인 필요 항목

- Architect 또는 System Keeper가 이어받아야 할 구조 이슈를 적습니다.
- 화면 책임, 페이지 분리, 레이아웃 경계, 폴더 책임, 토큰 관리 등을 포함할 수 있습니다.

예시 형식:

- 구직자 / 기업 인증 전 / 기업 인증 후 상태별 라우트 분리 방식
- 탐색 -> 후보 저장 -> 제안 -> 수락 흐름의 페이지 책임 분리
- 수락 전후 공개 레벨 차이를 구조적으로 반영하는 방식
- 공통 레이아웃과 페이지 레이아웃 분리 기준
- 브레이크포인트 토큰 적용 위치
- pages / components / layouts 책임 분리 점검

---

## 12. 콘텐츠 / 데이터 확인 필요 항목

- UX Writer나 Data Modeler가 이어받아야 할 항목을 적습니다.
- 운영형 문구, 상태 라벨, 카드 제목, 메타 정보, 더미 데이터 문맥 같은 내용을 분리해서 적습니다.

예시 형식:

- 설명형 문단이 운영형 상태 언어보다 먼저 보이는 문제
- 카드 제목 길이 과밀
- 버튼 라벨이 소개형 문구처럼 보이는 문제
- 샘플/예시처럼 읽히는 더미 데이터
- 카드 간 정보 비중이 모두 같아 보이는 문제
- 실제 운영 객체처럼 보이도록 상태값 재구성 필요

---

## 13. 시각 디자인 확인 필요 항목

- Hell-Visual Designer가 이어받아야 할 시각 이슈를 적습니다.
- 로고, 아이콘, 배지, 태그, 상태칩, 시각 리듬, MVP 인상 관련 항목을 분리해서 적습니다.

예시 형식:

- 카드 반복으로 인한 MVP 초안 인상
- 섹션별 시각 리듬 차이 부족
- 상태 배지/태그 스타일 통일성
- 아이콘 또는 그래픽 모티프 필요 여부
- 시각적 강약 (강조/보조/종속) 구분 미흡
- 로고 또는 배지 방향 미확정

---

## 14. 검수 기준 또는 실패 컷 관련 메모

- Responsive Auditor 또는 QA가 반드시 봐야 하는 항목을 적습니다.
- 이미 실패 판정이 난 항목이 있으면 번호나 조건을 함께 적습니다.
- 반복 실패 이슈가 있으면 재에스컬레이션 여부를 남깁니다.

예시 형식:

- 카드 제목, 상태 라벨, 메타 정보, 버튼 라벨 2줄 이상 내려가는 문제 재확인 필요
- 첫 화면 설명형 섹션이 운영 객체보다 먼저 보이는 문제 재검수 필요
- 동일 비중 카드 반복으로 시안 보드처럼 보이는 문제 재검수 필요
- 시각적 위계 부족으로 인한 MVP 인상 재검토 필요
- 동일 문제 반복으로 인한 재에스컬레이션 여부 확인 필요

---

## 15. 다음 담당자가 해야 할 일

- 다음 담당자를 명시합니다.
- 역할별로 해야 할 일을 분리해서 적습니다.
- "조금 더 다듬기"처럼 모호하게 쓰지 않습니다.
- 가능한 한 작업 지시가 바로 실행 가능한 문장이어야 합니다.

예시 형식:

- Hell-Architect:
  - `구직자`, `기업 인증 전`, `기업 인증 후` 상태를 기준으로 라우트와 접근 제어 구조를 설계할 것
  - `탐색 -> 구원 후보함 -> 구원 제안 -> 수락/거절 -> 수락 후 정보 공개` 흐름을 페이지 책임으로 나눌 것
- Hell-Security:
  - 목록 / 상세 / 제안 전 / 수락 후 공개 범위를 역할과 상태 기준으로 정리할 것
  - 희망 직무 기반 제한과 인증 상태 기반 제한을 분리해 문서화할 것
- Hell-Backend:
  - 공개 범위 차이를 API 응답 구조와 상태값 기준으로 정리할 것
  - 제안/수락/거절 흐름의 상태 전이를 명확히 설계할 것
- Hell-Builder:
  - 구조 설계 이후 공고 중심 UI나 일반 공개형 프로필 열람 구조로 되돌리지 말 것
- Hell-UX Writer:
  - 운영형 상태 언어와 버튼 라벨을 실제 서비스 문맥으로 정리할 것
- Hell-Visual Designer:
  - 섹션별 시각 리듬과 카드 반복으로 인한 MVP 인상을 시각적으로 개선할 것
  - 로고, 아이콘, 배지, 상태칩 스타일을 정의할 것

---

## 16. 다음 담당자 지정

- 다음에 바로 이어서 작업할 주 담당자를 적습니다.
- 필요하면 보조 담당자도 함께 적습니다.

예시 형식:

- 다음 주 담당자: `Hell-Visual Designer`
- 보조 담당자: `Hell-Builder`

또는

- 다음 주 담당자: `Hell-Builder`

---

## 17. 완료 기준

- 다음 담당자가 원문 대화를 다시 찾지 않아도 현재 상태를 이해할 수 있다.
- 확정사항, 가정사항, 추가 확인 필요 항목이 분리되어 있다.
- 역할별로 이어받아야 할 작업이 모호하지 않다.
- 구조 문제, 문구 문제, 데이터 문제, 시각 문제, 보안 문제, 백엔드 문제, 검수 문제가 섞여 있지 않다.
- 재에스컬레이션이 필요한 경우 해당 사실이 명시되어 있다.
