# Agent Handoff Template

## 현재 작업 범위

- 이번 인계는 `지옥` 서비스의 최신 기획 내용을, 대화에서 나온 표현과 맥락을 최대한 보존한 상태로 문서에 반영한 결과를 기준으로 한다.
- 단순 요약이 아니라, 다음 담당자들이 원문 대화를 다시 찾지 않아도 될 정도로 페이지 역할, 기능 흐름, 우선순위, 확정사항, 가정사항, 추가 확인 필요 항목을 풀어서 남겼다.
- 핵심은 `채용공고 중심 서비스가 아니라 구직자 프로필 중심의 역방향 구인 플랫폼`이라는 점과, `아무나 자기소개서를 볼 수 없도록 프라이버시와 인증 구조를 서비스 핵심으로 둔다`는 점이다.

## 유지해야 할 결정사항

- 서비스는 `직장을 올리는 사이트`가 아니라 `구직자가 자기 프로필(자기소개서)을 올리는 사이트`다.
- 기업은 채용공고를 앞세우기보다 사람을 먼저 탐색하고, 적합한 사람을 찾으면 `구원 제안`을 보낸다.
- 구직자는 `구원이 내려왔다`는 성격의 알림을 받고 수락 여부를 결정한다.
- 수락하면 직장 정보가 제공되는 흐름을 유지한다.
- 저장 기능의 감성 명칭은 현재 `구원 후보함`으로 유지한다.
- 아무 사용자나 구직자의 자기소개서를 열람할 수 없도록 기업 인증 구조를 둔다.
- 열람 가능 대상은 우선 `인사팀`과 `인사팀이 없는 중소기업의 대표자`를 기준으로 유지한다.
- 구직자 프로필 노출 범위는 `희망 직무 관련 인증 기업만`을 기본 방향으로 유지한다.
- 구직자의 과거 경력보다 `희망 직무`, `원하는 직장 방향`, `앞으로 가고 싶은 방향`을 더 중요하게 다룬다.
- 초기 안전장치 우선 포함 후보는 `열람 이력`과 `차단 기능`이다.

## 가정으로 남겨둔 부분

- 제안 알림의 실제 전달 채널은 아직 미확정이다. 실제 카카오 알림톡일 수도 있고, 내부 알림에 카카오톡 같은 경험을 입힐 수도 있다.
- 제안 수락 전 기업 정보 공개 범위는 미확정이다. 비공개/일부 공개/전체 공개 중 후속 결정이 필요하다.
- 팀장급 또는 실무 책임자까지 열람 권한을 확대할지 미확정이다.
- 외부 채용대행사 또는 헤드헌터를 허용할지 미확정이다.
- 프로필 상세 노출 범위와 마스킹 정책은 후속 구조 설계와 정책 설계에서 더 정교화가 필요하다.

## 다음 담당자가 해야 할 일

- Hell-Architect: `구직자`, `기업`, `기업 인증 전`, `기업 인증 후` 상태를 기준으로 라우트와 접근 제어 구조를 설계할 것.
- Hell-Architect: `탐색 -> 구원 후보함 -> 구원 제안 -> 수락/거절 -> 수락 후 정보 공개` 흐름을 페이지 책임으로 나눌 것.
- Hell-Architect: 희망 직무 기반 제한 노출이 목록, 상세, 제안 단계에 어떻게 걸리는지 구조적으로 정리할 것.
- Hell-Architect: 수락 전후 기업 정보 공개 레벨 차이를 구조에 반영할 수 있게 설계 포인트를 남길 것.
- Hell-Builder: 구조 설계 이후에도 공고 중심 UI나 일반 공개형 프로필 열람 구조로 되돌리지 말 것.
- Hell-Concept: `구원`, `지옥`, `내려온 구원` 같은 표현을 살리되 기능 의미를 흐리지 않는 카피 가이드를 정리할 것.

## 추가 확인 필요 항목

- 수락 전 회사명과 회사 정보 공개 범위
- 팀장급 채용 권한자의 허용 여부
- 외부 채용대행사 허용 여부
- 알림 전달 채널
- 공개 범위 세분화 수준

## 완료 기준

- 다음 담당자가 원문 대화를 다시 찾지 않아도 핵심 기획 흐름을 이해할 수 있다.
- 확정사항과 가정사항과 추가 확인 필요 항목이 분리되어 있다.
- Architect가 페이지 구조, 접근 제어, 인증 흐름, 제안 흐름을 추가 추측 없이 설계할 수 있다.
