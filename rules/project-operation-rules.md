# Project Operation Rules

## 목적

- 여러 에이전트가 같은 프로젝트를 순서대로 작업할 때 공통 실행 형식을 고정합니다.

## 규칙

1. 모든 작업은 아래 순서로 출력합니다.
   - 현재 이해한 조건
   - 판단 또는 결과
   - 이유
   - 다음 담당자에게 넘길 내용
2. 근거가 부족한 내용은 `가정`으로 분리합니다.
3. 기존 확정사항을 바꿀 때는 변경 이유를 먼저 적습니다.
4. 작업 단위는 한 번에 너무 넓게 잡지 않습니다.
5. 구조 문제는 Hell-Architect, 구현 문제는 Hell-Builder, 콘셉트 문제는 Hell-Concept, 범위 문제는 Hell-Planner, 검수 문제는 Hell-QA로 돌립니다.
6. 작업이 끝나면 인계 템플릿 또는 로그 파일에 상태를 남깁니다.
7. 사용자가 이미 구체적으로 정리한 기획, 구조, 흐름 설명은 과도하게 축약하지 않습니다.
8. 요약보다 정보 보존이 중요한 단계에서는 원래 항목 구조와 표현 결을 가능한 한 유지합니다.
9. 다음 담당자가 실제 작업에 필요한 맥락이 사라질 정도의 압축 정리는 금지합니다.
10. 페이지 역할, 기능 흐름, 조건, 예외, 우선순위, 확정사항, 가정사항은 한 줄 요약으로 뭉개지 말고 분리해 기록합니다.
11. 애매한 내용은 삭제하거나 축약하지 않고, `추가 확인 필요` 또는 `가정`으로 남깁니다.
12. 메인 화면은 소개용 랜딩으로 다루지 않으며, 운영형 상태 허브로 취급합니다.
13. 메인에 설명 섹션을 나열하는 대신 최근 등록 프로필, 최근 제안, 열람 가능 상태, 역할별 진입 액션 같은 운영 객체를 우선 배치합니다.
14. 서비스 구조와 사용자 흐름을 설명하는 문단형 콘텐츠는 메인에서 확장하지 말고 서비스 소개 페이지로 이동시키는 것을 기본 처리로 삼습니다.
15. Builder는 새 화면을 만들기 전에 그 화면이 `소개용`인지 `운영형`인지 먼저 판별해야 하며, 운영형 화면이면 객체/상태/행동 중심으로 구성해야 합니다.
16. Hell-QA는 결과물이 실패 컷 규칙에 걸리는지 먼저 판정한 뒤 세부 검수 내용을 작성해야 합니다.
17. Hell-QA는 추상 평가만 남기는 것을 금지하며, 모든 문제를 `위치 + 이유 + 수정 지시`로 함께 기록해야 합니다.
18. Hell-Builder는 QA 지적사항을 반영하면서 구조를 임의로 변경하는 것을 금지합니다.
19. Hell-Builder는 실패 컷 규칙에 해당하는 항목이 남아 있는 상태에서 `완료`로 보고할 수 없습니다.
20. 운영형 화면의 텍스트 길이, 문구 과밀, 상태 언어 부족 문제는 구현 미세 조정이 아니라 통과 여부를 가르는 강제 품질 기준으로 취급합니다.
21. Hell-Builder는 사용자가 별도로 중지 요청을 하지 않았다면 작업 종료 시 개발 서버를 직접 실행해 확인 가능한 상태로 둡니다.
22. 기본 개발 서버 실행 명령은 `npm run dev`이며, 다른 명령을 쓰는 경우에는 이유를 함께 기록합니다.

## 완료 기준

- 출력 형식이 고정된 순서를 따르는가
- 변경 사유 없이 기존 결정을 뒤집지 않았는가
- 다음 담당자가 바로 이어받을 수 있는가
- 실패 컷 규칙을 통과하지 못한 결과물을 완료 처리하지 않았는가
