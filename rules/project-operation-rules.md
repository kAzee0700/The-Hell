# Project Operation Rules

## 목적

- Hell-Orchestrator가 요청을 해석하고 적절한 담당자를 분기하는 기준을 정의합니다.
- 여러 에이전트가 순서대로 작업할 때 공통 실행 형식을 고정합니다.

## 역할 분기 기준

Hell-Orchestrator는 다음 기준에 따라 담당자를 선택합니다.

| 문제 유형 | 담당자 |
|-----------|--------|
| 목표, 범위, 흐름, 우선순위 애매 | Hell-Planner |
| 정보 구조, 라우팅, 레이아웃, 페이지 책임 | Hell-Architect |
| 문구, 카드 제목, 상태 라벨, 버튼 라벨, 안내 문구 | Hell-UX Writer |
| 더미 데이터, 상태값, 데이터 문맥 | Hell-Data Modeler |
| 시각 자산, 로고, 아이콘, 배지, 태그, 상태칩, 섹션 시각 리듬, MVP 인상 완화 | Hell-Visual Designer |
| React 컴포넌트, 레이아웃, 라우팅 구현 | Hell-Builder |
| 폴더 책임, 토큰 관리, 구조 중복, 과도한 추상화 | Hell-System Keeper |
| 반응형 품질, 오버플로우, 카드 위계, 브레이크포인트 | Hell-Responsive Auditor |
| UX 흐름, 접근성, 구조 준수, 실패 컷 판정 | Hell-QA |
| 인증, 권한, 열람 제한, 마스킹, 열람 이력, 차단 | Hell-Security |
| API, 데이터 모델, 상태 전이, 세션, 알림 | Hell-Backend |

## 역할 경계 구분

| 영역 | 담당자 | 하지 않는 일 |
|------|--------|-------------|
| 문구/라벨/운영형 인터페이스 언어 | Hell-UX Writer | 구조 변경, 시각 디자인 |
| 데이터 문맥/더미 데이터/상태값 | Hell-Data Modeler | 문구 재작성, 구조 변경 |
| 시각 자산/로고/아이콘/배지/시각 리듬/MVP 인상 완화 | Hell-Visual Designer | 구조 변경, 라우팅 변경, 문구 재작성 |
| 정보 구조/라우팅/레이아웃 | Hell-Architect | 문구 확정, 시각 디자인 결정 |
| React 구현 | Hell-Builder | 구조 임의 변경, 보안 정책 완화 |

## 구조 문제 vs 기타 문제 분기

- 구조 문제는 Hell-Architect가 먼저 판단합니다. Hell-Builder가 단독 처리하지 않습니다.
- 문구 문제는 Hell-UX Writer가 먼저 판단합니다. Hell-Builder가 라벨을 임의 확정하지 않습니다.
- 데이터 문맥 문제는 Hell-Data Modeler가 먼저 판단합니다.
- 시각 자산/시각 리듬 문제는 Hell-Visual Designer가 먼저 판단합니다.
- 반응형 품질 문제는 Hell-Responsive Auditor가 먼저 검수합니다.
- 보안/권한 문제는 Hell-Security가 먼저 판단합니다.
- 백엔드/API 문제는 Hell-Backend가 먼저 설계합니다.

## 작업 단위 원칙

1. 사용자 요청이 작고 명확하면 단일 에이전트만 호출합니다.
2. 여러 단계가 필요해도 최소 단계만 호출합니다.
3. 이미 QA까지 끝난 작업은 다시 Planner부터 돌리지 않습니다. 현재 상태에서 가장 적절한 역할만 고릅니다.
4. 한 번에 너무 많은 에이전트를 동시에 호출하지 않습니다.

## 출력 형식

모든 작업은 아래 순서로 출력합니다.

1. 현재 이해한 조건
2. 판단 또는 결과
3. 이유
4. 다음 담당자에게 넘길 내용

## 인계 및 갱신

1. 작업이 끝나면 인계 템플릿 또는 로그 파일에 상태를 남깁니다.
2. 구조/정책/UI/시각/보안/API 변경이 발생하면 `docs/changes/`에 변경 기록을 남깁니다.
3. `issue-board.md`, `error-log.md`, `unknowns-log.md`를 필요시 함께 갱신합니다.

## Builder 추가 기준

1. 새 화면을 만들기 전에 `소개용`인지 `운영형`인지 먼저 판별합니다. 운영형이면 객체/상태/행동 중심으로 구성합니다.
2. QA 지적사항을 반영하면서 구조를 임의로 변경하는 것을 금지합니다.
3. 실패 컷 규칙에 해당하는 항목이 남아 있는 상태에서 `완료`로 보고할 수 없습니다.
4. 사용자가 별도로 중지 요청을 하지 않았다면 작업 종료 시 개발 서버를 실행해 확인 가능한 상태로 둡니다.
5. 기본 개발 서버 실행 명령은 `npm run dev`이며, 다른 명령을 쓰는 경우에는 이유를 함께 기록합니다.

## QA 추가 기준

1. 결과물이 실패 컷 규칙에 걸리는지 먼저 판정한 뒤 세부 검수 내용을 작성합니다.
2. 추상 평가만 남기는 것을 금지합니다.
3. 모든 문제를 `위치 + 이유 + 수정 지시`로 함께 기록합니다.

## 완료 기준

- 출력 형식이 고정된 순서를 따르는가
- 역할 분기가 명확한가
- 변경 사유 없이 기존 결정을 뒤집지 않았는가
- 다음 담당자가 바로 이어받을 수 있는가
- 실패 컷 규칙을 통과하지 못한 결과물을 완료 처리하지 않았는가
