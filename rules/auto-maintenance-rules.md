# Auto Maintenance Rules

## 목적

- 에이전트가 로그와 인계 문서를 보고 다음 행동을 자동에 가깝게 판단할 수 있도록 기준을 고정합니다.

## 주의

- 실제 자동 실행 도구를 붙이는 문서는 아닙니다.
- 이 문서는 각 에이전트가 로그를 읽고 스스로 다음 작업을 선택하도록 돕는 운영 규칙입니다.

## 공통 흐름

1. 작업 시작 전 `issue-board.md`를 확인합니다.
2. `error`가 `open` 또는 `in-progress`면 Hell-Builder 또는 Hell-QA가 우선 봅니다.
3. `unknown`이 `high` 우선순위면 Hell-Planner 또는 Hell-Architect가 먼저 정리합니다.
4. 구조 변경이 필요한 오류는 Hell-Architect로 되돌립니다.
5. 표현 문제는 Hell-Concept가 문구 또는 스타일 기준을 정리합니다.
6. 수정이 끝나면 `issue-board.md`, `error-log.md`, `unknowns-log.md`, `handoff-template.md`를 함께 갱신합니다.
7. 같은 화면이 2회 연속 `시안 같다`, `소개형 같다`, `운영형처럼 안 보인다` 판정을 받으면 Builder 단독 수정으로 계속 진행하지 않습니다.
8. 텍스트 길이, 문구 과밀, 상태 언어 부족 문제가 2회 이상 반복되면 Hell-Concept를 다시 호출해 문구 축약과 라벨 재정의를 먼저 수행합니다.
9. 화면 위계, 카드 반복 구조, 첫 화면 정보 배치 문제가 2회 이상 반복되면 Hell-Architect를 다시 호출해 구조 재판단을 먼저 수행합니다.
10. QA가 같은 문제를 2회 이상 반복 기록했는데 해결되지 않으면 `issue-board.md`에 반복 실패 이슈로 별도 등록합니다.
11. 반복 실패 이슈가 등록되면 다음 담당자를 다시 지정하고, handoff에 `동일 문제 반복으로 인한 재에스컬레이션`을 반드시 명시합니다.
12. 반복 실패 상태에서는 `조금 더 다듬기` 같은 표현으로 넘기지 않고, 통과하지 못한 실패 컷 항목 번호를 함께 기록합니다.

## 담당자별 기본 반응

### Hell-Planner
- `unknown` 유형 중 기획 또는 범위 문제를 우선 확인합니다.
- 기능 추가 요청이 현재 목표를 벗어나면 보류 여부를 기록합니다.

### Hell-Architect
- 구조 변경 요청, 레이아웃 충돌, 책임 경계 문제를 확인합니다.
- Builder가 독단적으로 바꾸면 구조 판단을 다시 고정합니다.

### Hell-Builder
- `error` 유형 중 실제 수정 가능한 항목을 우선 처리합니다.
- 수정 전후 파일과 이유를 인계 문서에 남깁니다.
- 같은 품질 문제가 2회 이상 반복되면 단독 수정으로 계속 밀지 않고 Hell-Concept 또는 Hell-Architect 재분기 조건을 먼저 확인합니다.

### Hell-QA
- `resolved` 처리 전 재현 여부를 다시 확인합니다.
- 새 오류를 찾으면 `error-log.md`와 `issue-board.md`에 등록합니다.
- 같은 문제를 2회 이상 반복 기록했는데 해결되지 않으면 반복 실패 이슈로 등록하고 실패 컷 항목 번호를 함께 남깁니다.

### Hell-Concept
- 문구 충돌, 톤 불일치, 스타일 표현 혼선을 정리합니다.
- 구조를 바꾸지 않고 해결 가능한지 먼저 판단합니다.

## 완료 기준

- 열린 오류와 미확정 항목이 보드에서 추적 가능한가
- 어떤 담당자가 먼저 봐야 하는지 문서만 보고 판단 가능한가
- 수정 후 로그와 인계 문서가 함께 갱신되는가
- 반복 실패 이슈가 재에스컬레이션 규칙에 따라 분기되었는가
