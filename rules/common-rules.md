# Common Rules

## 목적

- 모든 에이전트가 공통으로 지켜야 하는 판단 원칙을 정의합니다.
- 역할 분기 규칙은 `project-operation-rules.md`에서 별도로 관리합니다.

## 공통 판단 원칙

1. 추측하지 않습니다. 근거가 없으면 `가정`으로 표시합니다.
2. 기존 결정사항을 임의로 뒤집지 않습니다.
3. 모든 제안에는 짧게라도 이유를 함께 적습니다.
4. 민감 정보 공개 범위는 추측하지 않고 정책 문서 또는 Hell-Security 판단을 따릅니다.
5. API 응답 구조와 상태값은 추측하지 않고 Hell-Backend 판단을 따릅니다.
6. 보안 정책과 편의성이 충돌하면 보안 정책을 우선합니다.
7. 백엔드 계약과 프론트 편의가 충돌하면 백엔드 계약을 우선합니다.

## 정보 보존 원칙

1. 사용자 요구사항, 현재 상태, 이전 결정사항을 먼저 요약합니다.
2. 사용자 이미 구체적으로 정리한 기획, 구조, 흐름 설명은 과도하게 축약하지 않습니다.
3. 요약보다 정보 보존이 중요한 단계에서는 원래 항목 구조와 표현 결을 가능한 한 유지합니다.
4. 다음 담당자가 실제 작업에 필요한 맥락이 사라질 정도의 압축 정리는 금지합니다.
5. 페이지 역할, 기능 흐름, 조건, 예외, 우선순위, 확정사항, 가정사항은 한 줄 요약으로 뭉개지 말고 분리해 기록합니다.
6. 애매한 내용은 삭제하거나 축약하지 않고, `추가 확인 필요` 또는 `가정`으로 남깁니다.

## 반복 실수 방지 원칙 (강제)

1. **같은 문제가 2회 이상 반복되면 완료 처리하지 않습니다.**
2. 수정 후 문제가 재발하면 원인을 분석하고 다시 수정합니다.
3. "조금 더 다듬기", "대체로 완료", "일단 넘어감" 같은 표현으로 반복 문제를 덮지 않습니다.
4. 반복되는 문제는 실패 컷 항목 번호와 함께 기록합니다.
5. 같은 문제가 3회 이상 반복되면 구조적으로 접근 방식을 바꿔야 합니다.

## 구현 품질 원칙 (강제)

1. **모든 컴포넌트와 섹션에는 반드시 기본 패딩이 포함되어야 합니다.**
2. **모든 텍스트와 콘텐츠에는 명시적인 정렬이 지정되어야 합니다.**
3. 패딩 없이 텍스트만 두는 것은 금지입니다.
4. `text-align`을 지정하지 않으면 기본 정렬에 의존하지 않습니다.
5. 패딩이 부족하면 축소하지 말고 증가시킵니다.

## 운영형 품질 강제 기준

아래 항목에 해당하면 결과물을 통과로 간주하지 않습니다.

1. 메인 화면의 카드 제목, 상태 라벨, 메타 정보, 버튼 라벨이 2줄 이상 내려가는 경우
2. 긴 문구를 줄바꿈으로 처리한 뒤 그대로 유지하는 경우
3. 카드 하나에 핵심 정보가 4개를 넘거나 설명문처럼 긴 문장이 들어가는 경우
4. 첫 화면에서 설명형 문단이나 서비스 소개 문장이 실제 상태, 최근 활동, 객체, 액션보다 먼저 시선을 차지하는 경우
5. 동일한 비중의 카드가 반복되어 샘플 보드, 시안 보드, 가이드 보드처럼 보이는 경우
6. 더미 데이터가 실제 운영 객체가 아니라 예시 카드, 샘플 데이터, 가이드 블록처럼 읽히는 경우
7. 기획서형 섹션 라벨이나 발표형 범주명이 실제 서비스 문맥을 해치는 경우
8. 운영형 상태 언어보다 설명형 문장이 앞서는 경우
9. **컴포넌트에 기본 패딩이 없는 경우**
10. **텍스트와 콘텐츠에 명시적인 정렬이 지정되지 않은 경우**

## 출력 원칙

1. 결과는 실행 가능한 문장, 구조, 코드, 체크리스트 형태로 정리합니다.
2. 작업 완료 후 다음 담당자에게 넘길 내용을 명시합니다.
3. 과도한 기능 추가는 하지 않습니다.

## 완료 기준

- 작업 결과가 모호한 표현 없이 실행 가능한 형태로 정리되어 있는가
- 가정과 확정사항이 분리되어 있는가
- 실패 컷 규칙에 해당하는 항목이 남아 있지 않은가
- 보안/백엔드 정책과 충돌하지 않는가
- 같은 문제가 2회 이상 반복되지 않았는가
- 기본 패딩과 정렬이 포함되어 있는가
