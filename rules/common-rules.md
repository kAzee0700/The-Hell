# Common Rules

## 목적

- 프로젝트 전반의 공통 작업 기준을 통일합니다.

## 규칙

1. 추측하지 않습니다. 근거가 없으면 `가정`으로 표시합니다.
2. 사용자 요구사항, 현재 상태, 이전 결정사항을 먼저 요약합니다.
3. 기존 결정을 임의로 뒤집지 않습니다.
4. 모든 제안에는 이유를 짧게 함께 적습니다.
5. 과도한 기능 추가를 하지 않습니다.
6. 결과는 실행 가능한 문장, 구조, 코드, 체크리스트 형태로 정리합니다.
7. 작업 완료 후 다음 담당자에게 넘길 내용을 명시합니다.
8. 사용자가 이미 구체적으로 정리한 기획, 구조, 흐름 설명은 과도하게 축약하지 않습니다.
9. 요약보다 정보 보존이 중요한 단계에서는 원래 항목 구조와 표현 결을 가능한 한 유지합니다.
10. 다음 담당자가 실제 작업에 필요한 맥락이 사라질 정도의 압축 정리는 금지합니다.
11. 페이지 역할, 기능 흐름, 조건, 예외, 우선순위, 확정사항, 가정사항은 한 줄 요약으로 뭉개지 말고 분리해 기록합니다.
12. 애매한 내용은 삭제하거나 축약하지 않고, `추가 확인 필요` 또는 `가정`으로 남깁니다.
13. 메인 화면은 소개 페이지처럼 보이면 안 되며, 실제 상태, 최근 활동, 객체, 액션이 서비스 설명보다 먼저 보여야 합니다.
14. 메인 화면에 사용자 흐름이나 서비스 구조를 길게 소개하는 콘텐츠를 두지 않으며, 해당 설명성 콘텐츠는 서비스 소개 페이지로 이동시키는 것을 기본 원칙으로 합니다.
15. 운영형 화면에서는 텍스트 설명보다 객체, 상태, 행동이 먼저 보여야 하며, 카드와 리스트는 실제 운영 데이터처럼 읽혀야 합니다.
16. 더미 데이터도 `예시`, `샘플`, `가이드`처럼 보이면 안 되며, 실제 등록 프로필, 실제 제안 상태, 실제 탐색 조건처럼 구성해야 합니다.
17. 메인 화면의 카드 제목, 상태 라벨, 메타 정보, 버튼 라벨이 2줄 이상 내려가면 결과물을 통과로 간주하지 않습니다.
18. 긴 문구를 줄바꿈으로 처리한 뒤 그대로 유지하는 방식으로 운영형 품질 문제를 덮는 것을 금지합니다.
19. 카드 하나에 핵심 정보가 4개를 넘거나 설명문처럼 긴 문장이 들어가면 결과물을 통과로 간주하지 않습니다.
20. 첫 화면에서 설명형 문단이나 서비스 소개 문장이 실제 상태, 최근 활동, 객체, 액션보다 먼저 시선을 차지하면 결과물을 통과로 간주하지 않습니다.
21. 동일한 비중의 카드가 반복되어 샘플 보드, 시안 보드, 가이드 보드처럼 보이면 결과물을 통과로 간주하지 않습니다.
22. 더미 데이터가 실제 운영 객체가 아니라 예시 카드, 샘플 데이터, 가이드 블록처럼 읽히면 결과물을 통과로 간주하지 않습니다.
23. 기획서형 섹션 라벨이나 발표형 범주명이 실제 서비스 문맥을 해치면 결과물을 통과로 간주하지 않습니다.
24. 운영형 상태 언어보다 설명형 문장이 앞서는 경우 결과물을 통과로 간주하지 않습니다.
25. 사용자가 별도로 중지 요청을 하지 않았다면, 구현 작업이 끝난 뒤에는 개발 서버를 직접 실행해 현재 결과를 바로 확인할 수 있는 상태로 둡니다.
26. 기본 실행 명령은 `npm run dev`로 처리하며, 프로젝트 사정상 다른 실행 명령이 필요할 때만 그 이유를 함께 남깁니다.

## 완료 기준

- 작업 결과가 모호한 표현 없이 실행 가능한 형태로 정리되어 있는가
- 가정과 확정사항이 분리되어 있는가
- 실패 컷 규칙에 해당하는 항목이 남아 있지 않은가
