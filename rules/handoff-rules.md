# Handoff Rules

## 목적

- 여러 에이전트가 같은 프로젝트를 이어서 작업할 때 충돌을 줄입니다.
- 다음 담당자가 원문 대화를 다시 찾지 않아도 현재 상태를 이해할 수 있도록 합니다.

## 기본 원칙

1. 역할 밖 판단은 하지 않습니다. 해당 담당자에게 넘깁니다.
2. 구조 변경이 필요하면 Hell-Architect로 되돌립니다.
3. 문구/라벨 문제가 있으면 Hell-UX Writer를 호출합니다.
4. 데이터 문맥 문제가 있으면 Hell-Data Modeler를 호출합니다.
5. 시각 자산/시각 리듬/MVP 인상 문제가 있으면 Hell-Visual Designer를 호출합니다.
6. 구조 규칙 위반 문제가 있으면 Hell-System Keeper를 호출합니다.
7. 반응형 품질 문제가 있으면 Hell-Responsive Auditor를 호출합니다.
8. 보안/권한 문제가 있으면 Hell-Security를 호출합니다.
9. 백엔드/API 문제가 있으면 Hell-Backend를 호출합니다.
10. 검수 단계에서는 새 기능을 임의로 추가하지 않습니다.

## 인계 필수 필드

handoff에는 아래 항목을 반드시 포함합니다.

1. 현재 작업 범위
2. 변경 파일 목록
3. 유지해야 할 결정사항
4. 가정으로 남겨둔 부분
5. 보안 정책 확인 필요 항목
6. 백엔드 설계 확인 필요 항목
7. 구조 확인 필요 항목
8. 콘텐츠/데이터 확인 필요 항목
9. 시각 디자인 확인 필요 항목
10. 다음 담당자가 해야 할 일
11. 다음 담당자 지정

## 재에스컬레이션 관련

1. QA가 같은 문제를 2회 이상 반복 기록했는데 해결되지 않으면 handoff에 `동일 문제 반복으로 인한 재에스컬레이션`을 반드시 명시합니다.
2. 통과하지 못한 실패 컷 항목 번호를 함께 적습니다.
3. 구조 변경이 필요한 항목은 Builder 작업으로 넘기지 않고 `Architect 재판단 필요`로 분리합니다.
4. 문구 문제는 Builder로 넘기지 않고 `Hell-UX Writer 판단 필요`로 분리합니다.
5. 데이터 문맥 문제는 Builder로 넘기지 않고 `Hell-Data Modeler 판단 필요`로 분리합니다.
6. 시각 자산/시각 리듬 문제는 Builder로 넘기지 않고 `Hell-Visual Designer 판단 필요`로 분리합니다.

## 인계 체크리스트

- [ ] 현재 작업 범위가 적혀 있는가
- [ ] 변경 파일 목록이 적혀 있는가
- [ ] 유지해야 할 결정사항이 적혀 있는가
- [ ] 가정으로 남긴 항목이 적혀 있는가
- [ ] 보안 정책 확인 필요 항목이 적혀 있는가 (해당 시)
- [ ] 백엔드 설계 확인 필요 항목이 적혀 있는가 (해당 시)
- [ ] 구조 확인 필요 항목이 적혀 있는가 (해당 시)
- [ ] 콘텐츠/데이터 확인 필요 항목이 적혀 있는가 (해당 시)
- [ ] 시각 디자인 확인 필요 항목이 적혀 있는가 (해당 시)
- [ ] 다음 담당자 작업이 적혀 있는가
- [ ] 다음 담당자 지정이 되어 있는가
- [ ] 반복 실패 여부와 재에스컬레이션 문구가 필요한 경우 포함되어 있는가
- [ ] 실패 컷 항목 번호가 필요한 경우 포함되어 있는가

## 시각 디자인 확인 필요 항목

시각 문제가 의심될 때 포함하는 항목:

- 로고 방향 또는 스타일
- 아이콘 통일성
- 상태 배지/태그/상태칩 스타일
- 섹션별 시각 리듬
- 카드 반복으로 인한 MVP 인상
- 시각 위계 (강조/보조/종속 구분)
- 그래픽 모티프 필요 여부
- Builder가 바로 적용 가능한 시각 기준
