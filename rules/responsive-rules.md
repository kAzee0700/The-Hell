# Responsive Rules

## 목적

- 모든 작업을 반응형 전제로 진행합니다.
- Hell-Responsive Auditor가 반응형 품질을 검수하는 기준을 제공합니다.

## 기본 원칙

1. 기본 스타일은 모바일 기준으로 작성합니다.
2. 태블릿과 데스크톱 확장은 필요한 지점에서만 추가합니다.
3. PC 화면만 기준으로 레이아웃을 판단하지 않습니다.

## 레이아웃 원칙

1. 고정 폭보다 유동 폭, `grid`, `flex`, `clamp`를 우선합니다.
2. 텍스트 줄바꿈, 버튼 크기, 여백이 작은 화면에서도 무너지지 않아야 합니다.
3. 화면 크기 변화에 따라 간격과 타이포가 급격히 무너지지 않도록 설계합니다.

## 패딩 원칙 (강제)

1. **모든 컴포넌트와 섹션에는 반드시 기본 패딩이 포함되어야 합니다.**
2. 패딩 없이 텍스트만 두는 것은 금지입니다.
3. 카드, 버튼, 섹션, 리스트 아이템은 최소 `var(--space-4)` 이상의 패딩을 포함합니다.
4. 패딩이 없으면 콘텐츠가 경계에 붙어 읽기 어려워지므로 항상 확인합니다.
5. 패딩이 부족하다고 느껴지면 축소하지 말고 증가시킵니다.

## 정렬 원칙 (강제)

1. **모든 텍스트와 콘텐츠는 명시적인 정렬을 가져야 합니다.**
2. `text-align`을 지정하지 않으면 기본 정렬에 의존하지 않습니다.
3. 중앙 정렬, 좌측 정렬, 우측 정렬 중 의도를 명확히 합니다.
4. 카드 내 라벨과 값의 정렬이 일관되어야 합니다.

## 텍스트 원칙

1. 제목, 상태 라벨, 메타 정보, 액션 라벨은 가능한 한 한 줄 우선으로 설계합니다.
2. 긴 문구는 자동 줄바꿈에 기대지 않고 축약, 말줄임, 짧은 문구 치환을 먼저 검토합니다.
3. 텍스트 오버플로우는 단순 시각 문제가 아니라 운영 품질 문제로 간주합니다.

## 카드 원칙

1. 카드 하나에는 핵심 정보 3~4개를 넘기지 않으며, 설명문처럼 긴 문장을 넣지 않습니다.
2. 모든 카드가 같은 크기, 같은 비중, 같은 강조 수준으로 반복되면 안 됩니다.
3. 가장 중요한 영역 1개, 보조 핵심 영역 2~3개, 종속 영역의 강약이 화면에서 구분되어야 합니다.

## 라벨 원칙

1. `OVERVIEW`, `PROFILES`, `OFFERS`, `ENTRY`, `TRUST` 같은 기획서형 라벨은 실제 서비스 문맥에 맞게 축소하거나 제거합니다.
2. `방금 도착`, `검토 중`, `수락 대기`, `열람 가능`, `인증 완료`, `공개 범위 유지` 같은 운영형 상태 언어를 우선 사용합니다.

## Hell-Responsive Auditor 검수 기준

반응형 품질 검수 시 아래 항목을 우선 확인합니다.

1. 모바일 기준에서도 주요 콘텐츠가 잘리지 않는가
2. 카드 제목, 상태 라벨, 메타 정보, 버튼 라벨이 2줄 이상 내려가는 오버플로우가 있는가
3. 카드 강약과 위계가 구분되는가
4. 컴포넌트 내부에 임의 숫자 브레이크포인트가 반복 선언되어 있지 않은가
5. 브레이크포인트가 공통 토큰 기준을 따르는가
6. **모든 컴포넌트에 기본 패딩이 포함되어 있는가**
7. **텍스트와 콘텐츠에 명시적인 정렬이 지정되어 있는가**

## UX Writer / Data Modeler / Visual Designer 개입 기준

아래 문제는 Hell-Responsive Auditor 검수 후 해당 담당자에게 넘길 수 있습니다.

- 텍스트 오버플로우가 문구 과밀에서 비롯된 경우 → Hell-UX Writer
- 카드 강약이 데이터 비중 문제에서 비롯된 경우 → Hell-Data Modeler
- 기획서형 라벨이 실제 서비스 문맥을 해치는 경우 → Hell-UX Writer
- 시각적 완성도, 패딩/정렬/여백 문제 → Hell-Visual Designer

## 완료 기준

- 모바일 기준에서도 주요 콘텐츠가 잘리지 않는가
- 화면 크기 변화에 따라 간격과 타이포가 급격히 무너지지 않는가
- 한 줄 우선 원칙이 지켜지는가
- 카드 강약과 위계가 구분되는가
- **모든 컴포넌트에 기본 패딩이 포함되어 있는가**
- **모든 텍스트와 콘텐츠에 명시적인 정렬이 지정되어 있는가**
