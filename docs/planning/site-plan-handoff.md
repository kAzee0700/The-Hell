# Agent Handoff Template

## 현재 작업 범위

- `지옥` 서비스의 1차 기획 결과를 문서화했다.
- 프로젝트 목적, 핵심 사용자, 페이지 목록, 기능 우선순위, 확정사항, 가정 사항을 정리했다.
- Architect가 추가 추측 없이 구조 설계를 시작할 수 있도록 핵심 설계 포인트를 정리했다.

## 변경한 파일 목록

- `docs/planning/site-plan-handoff.md`

## 유지해야 할 결정사항

- 서비스는 채용공고 중심이 아니라 `구직자 프로필 중심` 역방향 구인 플랫폼이다.
- 구직자가 프로필과 자기소개서를 먼저 올리고, 기업이 이를 탐색해 제안하는 흐름을 유지한다.
- 기업이 보낸 제안은 `구원이 내려왔다`는 감성의 알림 경험으로 전달한다.
- 제안을 수락하면 구직자에게 직장 정보가 제공되는 흐름을 유지한다.
- 저장 기능의 명칭은 `구원 후보함`으로 유지한다.
- 아무 기업이나 프로필을 열람할 수 없고, 인증된 기업만 열람 가능해야 한다.
- 열람 가능 대상은 우선 `인사팀` 또는 `인사팀이 없는 중소기업의 대표자`를 기준으로 유지한다.
- 구직자 프로필 노출 범위는 `희망 직무 관련 인증 기업만`을 기본 방향으로 유지한다.
- 열람 제한 기준은 과거 경력보다 `구직자의 희망 직무`를 우선한다.
- 초기 안전장치 후보는 `열람 이력`과 `차단 기능`을 우선 검토한다.

## 가정으로 남겨둔 부분

- 알림이 실제 카카오 알림톡인지, 내부 알림인지 최종 확정되지 않았다.
- 수락 전 회사 정보 공개 범위는 최종 확정되지 않았다.
- 팀장급 채용 권한자까지 열람 권한을 확대할지 확정되지 않았다.
- 외부 채용대행사 허용 여부는 확정되지 않았다.
- 공개 범위 정책의 세부 단계는 후속 설계에서 구체화가 필요하다.

## 다음 담당자가 해야 할 일

- Hell-Architect: `구직자`, `기업 인증 전`, `기업 인증 후` 상태를 기준으로 라우트/레이아웃/접근 제어 구조를 설계할 것.
- Hell-Architect: `탐색 -> 구원 후보함 -> 구원 제안 -> 수락/거절 -> 정보 공개` 흐름을 페이지 구조로 분해할 것.
- Hell-Architect: 희망 직무 기반 열람 제한과 인증 상태 기반 접근 제한이 충돌 없이 적용되도록 구조를 정의할 것.
- Hell-Concept: `구원`, `지옥`, `내려온 구원` 등의 감성 용어를 유지하되 기능 의미가 흐려지지 않도록 카피 가이드를 정리할 것.

## 완료 기준

- Architect가 추가 기획 추측 없이 페이지 구조와 라우트 구조를 설계할 수 있다.
- 인증 전/후 접근 제한과 희망 직무 기반 열람 제한이 구조 수준에서 분리되어 있다.
- 제안 수락 전후 정보 공개 차이를 반영할 수 있는 설계 방향이 정리되어 있다.
