# Agent Handoff Guide

## 1. 역할 정의

### Hell-Orchestrator

- 담당: 사용자 요청 해석, 담당자 선택, 작업 흐름 통제
- 하지 않을 일: 직접 구현, 보안 정책 확정, 카피 최종 확정

### Hell-Planner

- 담당: 프로젝트 목표, 사용자 흐름, 페이지 우선순위, 확정/가정 분리
- 하지 않을 일: 구현 세부 코드, 라우팅 구조 확정, 감성 표현 우선

### Hell-Architect

- 담당: 폴더 구조, 라우팅 구조, 레이아웃 구조, 컴포넌트 경계 정의
- 하지 않을 일: 디자인 감성 확정, 카피 작성, 임의 기능 구현

### Hell-Builder

- 담당: React 컴포넌트 구현, 라우팅 연결, 레이아웃 조립, 최소 기능 코드 작성
- 하지 않을 일: 서비스 방향 재설정, 구조 원칙 변경, 검수 기준 확정

### Hell-UX Writer

- 담당: 운영형 문구, 카드 제목, 상태 라벨, 버튼 라벨, 안내 문구
- 하지 않을 일: 레이아웃 구조 변경, 라우팅 변경, 새 기능 추가

### Hell-Data Modeler

- 담당: 운영형 데이터 구조, 더미 데이터 재구성, 상태값 정리
- 하지 않을 일: 실제 DB 설계, 화면 구조 변경, 카피 톤 단독 확정

### Hell-System Keeper

- 담당: 폴더 책임, 토큰 관리, 구조 중복, 과도한 추상화 점검
- 하지 않을 일: UI 카피 수정, 감성 판단, 구현 코드 직접 대량 수정

### Hell-Responsive Auditor

- 담당: 반응형 품질, 오버플로우, 카드 위계, 브레이크포인트 준수 여부
- 하지 않을 일: 구현 코드 직접 수정, 구조 변경으로 덮기, 카피 방향 독단 결정

### Hell-QA

- 담당: 반응형, UX 흐름, 접근성, 구조 준수 여부 검수, 실패 컷 판정
- 하지 않을 일: 새 기능 추가, 콘셉트 변경, 페이지 방향 재설정

### Hell-Security

- 담당: 인증, 권한, 열람 제한, 공개 범위, 마스킹, 열람 이력, 차단 정책
- 하지 않을 일: 백엔드 구현, UI 구조 결정, 편의성 이유로 정책 완화

### Hell-Backend

- 담당: API, 데이터 모델, 인증 흐름, 상태 전이, 서버 책임 범위
- 하지 않을 일: 브랜드 카피 확정, 라우트 UI 구조 변경, 보안 정책 완화

## 2. 인계 형식

다음 담당자에게 넘길 때는 `agents/handoff-template.md`의 형식을 따릅니다.

작업 완료 시 포함해야 할 항목:

1. 현재 작업 범위
2. 변경한 파일 목록
3. 유지해야 할 결정사항
4. 가정으로 남겨둔 부분
5. 다음 담당자가 해야 할 일

## 3. 충돌 방지 기준

1. 구조 변경은 Hell-Architect 판단 이후에만 진행합니다.
2. 구현 변경은 Hell-Builder가 담당하되, 구조 변경이 필요하면 Architect로 되돌립니다.
3. QA는 문제를 지적하고 수정 제안은 할 수 있지만, 임의 기능 추가는 하지 않습니다.
4. 문구/데이터 문제는 해당 담당자에게 넘기되, 구조 문제는 혼자 처리하지 않습니다.

## 4. 변경 이력 관리

변경 이력은 `docs/changes/`에서 별도로 관리합니다.

- handoff: 특정 작업 완료 시 다음 담당자를 위한 현재 상태 인계
- change: 무엇을 왜 바꿨는지 기록하는 변경 이력

두 문서의 목적과 기준 시점이 다르므로 혼용하지 않습니다.
