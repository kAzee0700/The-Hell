# Hell Project Site Architecture Draft

## 1. 목적

- 이 문서는 `docs/planning/site-plan.md`에 확정된 기획을 React 라우트, 레이아웃, 페이지 책임, 접근 제어 구조로 옮기기 위한 초안이다.
- 중심 객체는 `채용공고`가 아니라 `구직자 프로필`이다.
- 구조 설계의 핵심은 `구직자`, `기업`, `기업 인증 전`, `기업 인증 후` 축을 분리하고, `희망 직무 기반 제한 노출`과 `구원 제안 흐름`을 구조 수준에서 반영하는 것이다.

## 2. 사용자 축 분리

### 2.1 공통 공개 사용자

- 아직 로그인하지 않았거나, 서비스 구조를 이해하려는 방문자다.
- 메인, 서비스 소개, 운영 정책 및 안전 안내 페이지에 접근할 수 있다.
- 구직자 탐색 결과나 프로필 상세는 접근할 수 없다.

### 2.2 구직자

- 자신의 프로필과 자기소개서를 등록하고 수정하는 사용자다.
- 자신이 받은 `구원 제안`을 확인하고 수락/거절하는 사용자다.
- 기업 탐색 페이지의 주 사용자 축은 아니다.

### 2.3 기업 인증 전 사용자

- 기업 계정은 있으나, 아직 사업자 및 채용 권한 인증이 끝나지 않은 상태다.
- 기업용 진입은 가능하지만, 탐색 목록, 프로필 상세, 후보 저장, 제안 발송은 막아야 한다.
- 이 상태에서는 `기업 인증 및 권한 확인 페이지`로 유도하는 구조가 필요하다.

### 2.4 기업 인증 후 사용자

- 인증된 기업의 인사팀, 채용 담당자, 인사팀이 없는 중소기업의 대표자를 우선 기준으로 둔다.
- 구직자 탐색, 프로필 상세 열람, 구원 후보함, 구원 제안 발송 흐름에 접근할 수 있다.
- 단, 모든 인증 기업이 모든 프로필을 보는 구조가 아니라 `희망 직무 기반 제한 노출`을 전제로 한다.

## 5. 레이아웃 책임 분리

### 5.1 `RootLayout`

- 전체 사이트 공통 골격을 담당한다.
- 공통 헤더, 공통 푸터, 기본 `main` 영역을 제공한다.
- 공개 페이지와 보호 페이지 모두의 최상위 껍데기 역할만 한다.
- 인증 여부나 역할별 내비게이션 판단은 여기서 과도하게 맡지 않는다.

### 5.2 `PublicLayout`

- 메인, 서비스 소개, 운영 정책 및 안전 안내처럼 공개 접근이 가능한 페이지 묶음을 담당한다.
- 서비스 소개형 헤더, 기본 안내 CTA, 공개 정보 영역을 포함한다.
- 기업 인증이 필요한 행동 버튼은 내부적으로 인증 페이지 유도로 연결될 수 있다.

### 5.3 `JobSeekerLayout`

- 구직자 프로필 등록/수정, 구직자 제안 확인 흐름을 담당한다.
- 구직자용 내비게이션과 상태 안내를 가진다.
- `내 프로필`, `받은 제안`, 향후 열람 이력/차단 기능이 추가될 여지를 남긴다.

### 5.4 `CompanyGateLayout`

- 기업 인증 전 상태를 처리하는 레이아웃이다.
- 탐색/상세/제안으로 바로 들어오려는 비인증 기업을 `인증 필요` 메시지와 함께 인증 페이지로 유도한다.
- 접근 거부와 전환 CTA를 같은 구조 안에서 다룬다.

### 5.5 `CompanyLayout`

- 인증된 기업 전용 탐색 흐름을 담당한다.
- `구직자 탐색`, `구원 후보함`, `보낸 제안` 같은 기업용 내비게이션을 포함한다.
- 이 레이아웃 아래 페이지들은 기본적으로 기업 인증 통과를 전제로 한다.

## 6. 페이지 책임 분리

### 6.1 메인 페이지

- 공개 진입점이다.
- `공고를 보는 사이트가 아니라 사람을 보는 사이트`라는 메시지를 첫 화면에서 이해시키는 역할만 맡는다.
- 실제 탐색 결과나 상세 프로필 데이터 책임은 가지지 않는다.

### 6.2 서비스 소개 페이지

- 서비스 철학, 사용 방식, 일반 채용사이트와의 차이를 설명한다.
- 구조 설명이 목적이며, 실제 보호 데이터 노출이나 사용자 상태 분기는 담당하지 않는다.

### 6.3 운영 정책 및 안전 안내 페이지

- 열람 가능 주체, 공개 범위 기본값, 희망 직무 기반 제한 노출, 열람 이력, 차단, 신고 정책을 설명한다.
- 실제 권한 판단은 하지 않지만, 서비스 정책의 기준 문서 역할을 한다.

### 6.4 구직자 프로필 등록/수정 페이지

- 구직자가 자기소개서, 기술/경험, 희망 직무, 원하는 방향, 공개 범위를 입력/수정하는 책임을 가진다.
- 구직자 데이터 생성과 수정 책임을 가진다.
- 기업 탐색 기능이나 제안 발송 책임은 갖지 않는다.

### 6.5 구직자용 제안 및 알림 페이지

- 받은 제안을 목록으로 보여주고, 제안 상세 확인과 수락/거절 흐름으로 연결한다.
- 알림 전달 채널은 구조 밖에 두고, 페이지는 `받은 제안 상태를 확인하고 행동하는 화면` 책임만 가진다.
- 수락 전후 기업 정보 노출 레벨 분기를 화면 구조에서 처리할 수 있어야 한다.

### 6.6 기업 인증 및 권한 확인 페이지

- 사업자 정보, 소속, 채용 권한 여부를 확인한다.
- 인증 완료 전 기업 사용자가 보호 페이지에 진입하려 할 때 전환 목적지 역할을 한다.
- 실제 탐색 결과나 프로필 데이터는 이 페이지에서 다루지 않는다.

### 6.7 구직자 탐색 페이지

- 인증 기업이 제한된 범위 안에서 프로필 목록을 탐색하는 책임을 가진다.
- 필터, 검색, 정렬, 저장 진입, 상세 진입을 담당한다.
- 카드에 보여줄 정보는 목록 수준으로 제한하고, 상세 판단은 프로필 상세 페이지로 넘긴다.

### 6.8 구직자 프로필 상세 페이지

- 기업이 한 명의 구직자를 깊게 읽고 `구원 제안` 여부를 판단하는 책임을 가진다.
- 자기소개서, 기술/경험, 희망 직무, 희망 조건, 저장 액션, 제안 액션을 포함한다.
- 목록 권한과 별개로 상세 접근 권한을 한 번 더 확인할 수 있어야 한다.

### 6.9 구원 후보함 페이지

- 기업이 저장한 후보를 다시 보고 비교하고 제안으로 이어가는 페이지다.
- 일반 저장 목록과 기능은 같지만, 서비스 용어는 `구원 후보함`으로 유지한다.
- 탐색 결과를 대체하는 페이지가 아니라, 탐색 이후 보관함 역할에 집중한다.

### 6.10 기업용 제안 발송 페이지

- 특정 구직자에게 제안을 작성하고 보내는 액션 책임을 가진다.
- 제안 대상 정보 확인, 제안 메시지 입력, 발송 동작을 다룬다.
- 알림 채널 확정 전까지는 `제안 생성` 책임만 가진다.

### 6.11 기업용 보낸 제안 확인 페이지

- 기업이 자신이 보낸 제안 목록과 상태를 확인하는 페이지다.
- 초기에는 최소 목록 수준으로 시작할 수 있다.
- 상세 상태 추적 고도화는 후속 우선순위로 남긴다.

## 7. 접근 제어가 필요한 지점

### 7.1 탐색 페이지 접근

- `/talents`는 인증된 기업만 접근 가능해야 한다.
- 인증 전 기업 또는 공개 사용자는 `/company/verify` 또는 공개 안내 페이지로 유도한다.
- 추가로 `희망 직무 기반 제한 노출`이 목록 데이터 단계에서 적용되어야 한다.

### 7.2 프로필 상세 접근

- `/talents/:talentId`는 인증된 기업만 접근 가능해야 한다.
- 목록 접근이 가능하더라도, 상세에서도 다시 `열람 가능한 희망 직무 범위인지` 확인할 수 있어야 한다.
- 후속 정책에 따라 일부 정보 마스킹이 필요할 수 있으므로, 상세 내부에서도 노출 레벨 분기를 고려한다.

### 7.3 구원 후보함 접근

- `/saved`는 인증된 기업만 접근 가능해야 한다.
- 저장 대상 역시 탐색/상세와 같은 권한 조건 안에서만 유지되어야 한다.

### 7.4 제안 발송 접근

- `/offers/new/:talentId`는 인증된 기업만 접근 가능해야 한다.
- 단순 인증 여부뿐 아니라, 해당 구직자를 제안 가능한 범위인지 확인해야 한다.
- 제안 발송 후 알림 전달 채널은 구조적으로 분리하고, 화면은 발송 성공 상태만 책임진다.

### 7.5 구직자 제안 확인 접근

- `/proposals`와 `/proposals/:proposalId`는 구직자 본인만 접근 가능해야 한다.
- 수락 전에는 제한된 기업 정보만 보이고, 수락 후에는 더 많은 정보가 열릴 수 있도록 상태 분기를 둔다.

### 7.6 정책 페이지 접근

- `/safety`는 공개 라우트로 둔다.
- 다만 여기의 설명과 실제 권한 판단 로직은 분리해야 한다.

## 8. 구원 후보함 / 정책 / 안전 안내 페이지의 위치와 경계

### 8.1 구원 후보함

- 기업 인증 후 영역 안에 둔다.
- 탐색 흐름의 하위 후속 페이지로 본다.
- 메인 공개 내비게이션에서 전면 노출하는 페이지가 아니라, 기업 사용자 작업 공간의 일부로 둔다.

### 8.2 운영 정책 및 안전 안내

- 공개 영역에 둔다.
- 구직자와 기업 모두가 로그인 전에도 확인할 수 있게 한다.
- 인증 구조, 제한 노출, 열람 이력, 차단 기능의 설명 기준점이므로, 부속 푸터 링크 수준을 넘어 독립 페이지로 유지한다.

## 10. Builder에게 넘길 구현 명세

### 10.1 라우트 우선 구현 순서

1. 공개 라우트부터 추가한다.
2. 구직자 프로필 등록/수정 페이지를 추가한다.
3. 기업 인증 페이지를 추가한다.
4. 기업 인증 후 라우트인 탐색 목록, 상세, 제안 발송 흐름을 추가한다.
5. 구직자 제안 확인 페이지를 추가한다.

### 10.2 초기 구현 시 반드시 지킬 점

- 현재 `RootLayout`는 유지하되, 하위 레이아웃을 분리해 역할별 책임을 나눈다.
- `pages`는 라우트 단위 화면만 만들고, 카드/폼/패널은 `components`로 분리한다.
- 인증 여부와 권한 여부는 우선 목 데이터 또는 임시 가드 구조로 분리하고, 실제 인증 연동 전에도 레이아웃 경계는 유지한다.
- `희망 직무 기반 제한 노출`은 실제 데이터 규칙이 완전히 확정되지 않았더라도, 목록/상세/제안 라우트에서 같은 접근 제어 훅 또는 가드 포인트를 사용할 수 있게 둔다.
- 알림 전달 채널은 아직 미확정이므로, `구직자 제안 확인 페이지`는 내부 제안 목록 UI로 구현하고 외부 채널 연동을 전제로 고정하지 않는다.
- `구원 후보함`은 별도 기업 라우트로 분리하되, 탐색 결과를 그대로 복제하지 말고 저장한 후보를 다시 보는 역할만 맡긴다.
- 수락 전 기업 정보 공개 범위는 미확정이므로, 제안 상세 화면은 `preAccept`와 `postAccept` 상태에 따라 노출 블록을 나눌 수 있는 구조로 만든다.

## 11. 가정으로 남기는 구조 포인트

- 알림 전달 채널은 라우트 밖 문제로 두고, 현재 구조에서는 `제안 알림 확인 페이지`만 확정한다.
- 수락 전 기업 정보 공개 범위는 `제한 공개` 상태가 존재한다는 전제만 구조에 둔다.
- 팀장급 채용 권한자와 외부 채용대행사 허용 여부는 현재 구조에 직접 박지 않고, 인증 정책 확장 포인트로 남긴다.
- 희망 직무 기반 제한 노출의 세부 매핑 규칙은 미확정이므로, 접근 제어 포인트만 공통화하고 구체 규칙은 후속 결정으로 남긴다.

## 12. 완료 기준

- Builder가 추가 추측 없이 라우트와 레이아웃을 구현할 수 있다.
- 공개 영역, 구직자 영역, 기업 인증 전 영역, 기업 인증 후 영역이 구조적으로 분리되어 있다.
- 탐색, 상세, 제안, 알림 흐름에서 접근 제어 지점이 드러난다.
- `구원 후보함`, `운영 정책 및 안전 안내`의 위치와 경계가 분명하다.
